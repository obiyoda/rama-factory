Problem

Dogfood generated apps without letting them leak into the factory root.

Generated apps are proof that seeds work. They need commit history, validation results, and agent attribution. They should be first-class seed labs, not accidental untracked folders beside the framework code.

See lifecycle Commit policy Command plan

Boundary

Four surfaces, four different commit rules.

The factory stays clean because each surface has a clear purpose. Dogfood work gets committed, but it commits to the correct surface.

Factory core

CLI, generator, validators, docs, skills, seed definitions, and tests. This is the framework repo.

Seed labs

Generated apps used to prove a seed. They are created from a manifest and keep their own commit evidence.

Examples

Curated snapshots that demonstrate known-good flows. They are smaller and more stable than active labs.

Business apps

Real products started from Rama Factory. They should live in their own repos and own their copied code.

Commit Policy

Commit evidence where it answers the right question.

The current untracked factory-floor/ folder is useful evidence, but its location hides intent. A seed lab makes the evidence explicit.

Question Commit Surface Rule
Did the seed generate a runnable app? Seed lab baseline commit Commit the untouched generated app before agent edits.
Did an agent improvement work? Seed lab app commit Commit each meaningful checkpoint with validation output in the lab report.
Should every future app get this change? Factory seed template commit Promote selected files back into factory/seeds/*.
Should open-source users see this demo? Tracked example snapshot Publish only after the lab is validated and cleaned.
Is this one business idea specific? Separate product repo Keep it outside the factory once it becomes a real app.

Data Model

The seed system gets a lab layer.

A lab manifest records what was generated, which seeds were installed, where the working app lives, and which commands prove it still works.

  .rama-workspaces/       # ignored
    factory-floor/        # generated app, own git repo

  docs/
    dogfood-seed-system.html

Lab manifest shape

factory/seed-labs/factory-floor.edn should be plain data so agents, tests, docs, and future Rama dashboards can read it.

  • :lab/id, :lab/title, and :lab/workspace
  • :app/name and starter generator version
  • :seeds with source paths and versions
  • :validation/commands for build and tests
  • :commits for baseline, checkpoints, and promotions

CLI Shape

Make dogfooding an explicit workflow.

These commands are the likely next slice. They make lab generation, checkpointing, validation, diffing, and promotion repeatable.

1

rama lab new factory-floor --seed factory-dashboard

Creates a lab manifest, generates the app under .rama-workspaces/, installs the seed, initializes a lab git repo, and creates the baseline commit.

Output: manifest + clean generated app
2

rama lab validate factory-floor

Runs the lab validation commands, captures output paths, and appends a validation event to the lab report.

Output: validation event
3

rama lab checkpoint factory-floor -m "Improve dashboard"

Commits the lab app state after validation and records persona, role, commit, and gate metadata.

Output: evidence commit
4

rama lab diff factory-floor --against seed

Shows a template-aware diff between the active lab app and the seed source it came from.

Output: promotion candidates
5

rama lab promote factory-floor --to factory-dashboard

Copies selected changes back into seed templates, preserving placeholders and requiring tests before commit.

Output: seed improvement
6

rama lab publish-example factory-floor

Writes or refreshes a curated examples/factory-floor snapshot after promotion decisions are made.

Output: public demo

Current State

What to do with the existing untracked factory-floor/.

It should not be committed at repo root as-is. It should become either a seed lab workspace or a cleaned example.

Recommended now

Move or regenerate it under .rama-workspaces/factory-floor, initialize a lab git repo, and commit a baseline after pnpm regeneration.

Promote later

When the dashboard seed improves inside the lab, promote only template-worthy changes back into factory/seeds/factory-dashboard.

Publish when stable

Once the lab is useful for open-source users, publish a small tracked snapshot under examples/factory-floor.

Decision

Build seed labs as part of the seed system. Keep active generated apps outside the root tree, but give them commit history, validation records, agent attribution, and an explicit promotion path back into seeds.
